# ADR-006: Search Engine Phasing

Status: Accepted

Source: `docs/package/codex/01_Adopted_Decisions_ADR.md` ADR-006. Trace: DEC-05, C-5.

## Context

Adding PostgreSQL FTS, OpenSearch, vector indexes, and graph storage too early would multiply permission-filter surfaces before the permission model is frozen.

## Decision

Use PostgreSQL 16 FTS for R3 keyword search. Decide OpenSearch only at the R3 Gate based on Korean-tokenization evidence. Vector search uses pgvector no earlier than R6. Neo4j and GraphSync are prohibited before R7. Search permissions must be injected at query time, not filtered after retrieval.

## Consequences

R3 infrastructure remains PostgreSQL-centered. Search quality tradeoffs are accepted until measured. Any later engine transition must re-prove permission filtering, tenant isolation, metadata leakage controls, and index synchronization.

## Review Triggers

Revisit at the R3 search evaluation gate, on measured PG FTS SLA failure, during R6 hybrid retrieval planning, and during R7 graph planning.

## R3 Gate Update - 2026-06-12

Status: Accepted for R3 technical completion under the active operator waiver for human approval.

Input:

- `docs/reports/R3_korean_tokenization_eval.md`
- `docs/reports/R3_opensearch_decision_proposal.md`

Measured PG FTS `simple` results on the synthetic Korean legal-term fixture: 30 cases, 90 snippets, 100.0% precision, 55.0% recall, 0.0% false-positive rate.

Update:

- PostgreSQL FTS remains the R3 shipped search engine.
- OpenSearch implementation, dependencies, and runtime services remain deferred after R3.
- The measured Korean recall gap is recorded as a future search-quality driver, not as approval to widen the R3 permission surface before query-time permission filters are complete.
- Any later OpenSearch transition must re-prove Permission-before-search, tenant isolation, metadata leakage controls, deleted/superseded exclusion, safe snippets/highlights, audit coverage, and fail-closed behavior on the new backend before user-facing traffic moves.
